# 面试准备

> 目标：把 Doc2Action 做成“能演示、能解释、能复盘”的求职项目。

---

## 一、项目 30 秒自我介绍

Doc2Action 是一个把非结构化文档（会议纪要、PRD、邮件）转成可执行结果的 AI 工作台。  
核心流程是：输入文档 -> 结构化提取 -> 引用回溯 -> 人工审核 -> 导出结果。  
我重点解决了三个问题：输出结构化、结果可追溯、人机协同可落地。

---

## 二、常见问题与答题要点

### 1) 你为什么做这个项目？

- 对齐真实工作场景：文档多、信息杂、行动项易遗漏
- 体现 AI 应用落地能力，而不是只做聊天功能
- 可覆盖产品思维 + 工程实现 + LLM 实战能力

### 2) 这个项目的 MVP 边界是什么？

- 必做：输入、结构化提取、引用依据、人工编辑、导出
- 不做：登录/协作/企业集成/自动执行
- 原则：先闭环再扩展，先可靠再智能

### 3) 技术方案为什么这么选？

- 前端：Next.js + TypeScript，开发效率高、适合作品展示
- 后端：FastAPI，便于处理文本与 LLM 输出 schema
- 存储：本地 JSON/SQLite，降低初期复杂度

### 4) 你如何控制 LLM 幻觉？

- 使用固定输出 schema
- 强化“基于输入文本”的提示词约束
- 结果展示引用 chunk，便于人工审核
- 高风险字段（owner/due date）先弱化为可选

### 5) 你做过哪些关键取舍？

- 放弃“看起来高级”的功能，优先打通最小闭环
- 放弃复杂自动化，保留人工确认环节
- 优先稳定和可解释，而非追求全自动

## 五、算法岗升级答题要点（Senior 导向）

### 1) 你如何证明自己有“算法能力”而不是只会调 API？

- 我做了三层能力：规则基线、API 模型、LoRA 微调模型
- 每层都有离线评估指标和误差分析，而不是只看主观体验
- 我能解释每次迭代为什么有效、代价是什么

### 2) 你如何设计训练与评估闭环？

- 数据：定义结构化标注 schema，构建 train/val/test
- 训练：LoRA/QLoRA 跑 SFT，记录超参与 loss
- 评估：字段级 precision/coverage + citation hit rate
- 迭代：按误差类型回补数据和 prompt/规则

### 3) 你如何把算法方案落到工程系统？

- 模型层与服务层解耦（可切换 rule/API/local model）
- 保留降级与重试策略，保证可用性
- 通过 sample checks + CI 做质量门禁

## 六、第 1 周可直接使用的面试问答清单

### Q1：`backend/app/main.py` 这条链路你怎么设计的？

**答题模板：**

- 我把链路拆成四层：输入清洗、语义切块、提取策略、统一响应。
- 入口是 `POST /analyze`，先做 `clean_text` 和 `split_into_chunks`，保证输入稳定且可追溯。
- 提取层支持 `rules/llm/auto` 模式，`auto` 下如果 LLM 失败会回退规则，保证服务可用性。
- 输出统一到 `AnalyzeResponse`，并保留 `chunks/meta` 用于可解释和评估。

### Q2：为什么不用“只靠 LLM”？

**答题模板：**

- 只靠 LLM 会有成本、网络、权限等不确定性。
- 我保留规则基线是为了兜底可用，同时也便于做 A/B 对比。
- 在离线评估里，我可以量化规则与 LLM 的差异，决策是否值得投入更多 token 或训练资源。

### Q3：你如何证明这个项目不是“拍脑袋优化”？

**答题模板：**

- 我用 `ml/eval/evaluate.py` 做字段级离线评估，不靠主观印象。
- 指标包含 `action/risk/question F1`、`summary_jaccard`、`citation_hit_rate`、`duplication_rate`。
- 每次改动后都产出对比报告，确保“改了就变好”是可验证的。

### Q4：你做数据精修时为什么先做 Jira Top30？

**答题模板：**

- 先做高优先样本而不是随机样本，能在同样时间内获得最大质量收益。
- Top30 的问题集中在 `summary 元数据化`、`action 过长`、`risk/question 缺失`，属于高频高影响错误。
- 我先用 LLM 生成草案，再人工复核，形成“效率 + 质量”平衡的数据闭环。

### Q5：如果面试官问“你真正的技术难点是什么”？

**答题模板：**

- 最大难点不是调用模型，而是把系统做成“可用、可解释、可评估”。
- 我通过三件事解决：双策略回退、引用追溯、离线指标体系。
- 这样做的价值是：即使模型变化，系统仍可稳定迭代，而不是一次性 demo。
