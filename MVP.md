## 1. 文档目的

本文件用于明确 **Doc2Action AI Workspace** 第一版（MVP）的功能边界，避免项目范围失控，确保在较短时间内完成一个**可演示、可测试、可写入简历**的最小闭环版本。

本文件主要回答三个问题：

1. 第一版到底要做什么
2. 第一版明确不做什么
3. 第一版做到什么程度算成功

---

## 2. MVP 核心目标

第一版的目标不是做一个完整 SaaS，也不是做一个复杂 AI Agent 平台，而是做出一个清晰的闭环：

**用户输入一段文档或文本 → 系统提取结构化执行信息 → 用户审核编辑 → 导出结果**

第一版重点验证以下价值：

- AI 是否能把非结构化内容转成执行结果
- 输出是否足够结构化、可读、可修改
- 是否能通过引用原文提升可信度
- 是否能形成一个完整可演示 workflow

---

## 3. MVP 用户场景

第一版只服务以下基础场景：

### 场景 1：会议纪要整理

输入会议纪要，输出：

- Summary
- Action Items
- Risks
- Open Questions

### 场景 2：需求文档整理

输入 PRD / 需求说明，输出：

- 摘要
- 开发任务草稿
- 风险点
- 待确认问题

### 场景 3：邮件 / 沟通记录整理

输入邮件链或聊天记录，输出：

- 结论摘要
- 待办事项
- 风险与遗漏点
- 下一步建议

---

## 4. MVP 必做功能（Must Have）

以下功能属于第一版必须完成的最小能力。

### 功能 1：文本输入

支持以下两种输入方式：

- 手动粘贴文本
- 上传文本类文件（建议先支持 `.txt` / `.md`）

### 目标

确保用户能快速把内容送进系统。

---

### 功能 2：文档预处理

系统对输入文本进行基础清洗和切分，包括：

- 去除空行
- 分段
- chunk 编号
- 基础结构整理

### 目标

为后续 LLM 提取和引用映射做准备。

---

### 功能 3：结构化结果生成

系统调用 LLM 生成以下四类结果：

- **Summary**
- **Action Items**
- **Risks**
- **Open Questions**

### 目标

让输出从“普通回答”变成“结构化执行信息”。

---

### 功能 4：引用原文依据

每条 Action Item / Risk / Open Question 尽量显示其来源原文片段或段落编号。

### 目标

提高结果可信度，减少“AI 乱编”的感觉。

---

### 功能 5：人工审核与编辑

用户可以对生成结果进行编辑，包括：

- 修改 summary
- 编辑 / 删除 action item
- 调整优先级
- 补充遗漏项

### 目标

保证系统不是直接替代人，而是辅助人完成确认。

---

### 功能 6：结果导出

支持导出至少一种结构化格式，建议优先：

- Markdown

可选补充：

- JSON
- CSV

### 目标

让结果能真正带走，而不只是停留在页面里。

---

## 5. MVP 可选功能（Nice to Have）

这些功能有价值，但第一版可以先不做，若时间充裕再考虑加入。

### 可选 1：文档类型选择

用户可以选择输入内容类型：

- Meeting Notes
- PRD
- SOP
- Email Thread

作用：

让 prompt 更精准，输出更稳定。

---

### 可选 2：优先级自动标注

系统为 action items 自动标注：

- High
- Medium
- Low

作用：

增强执行感，但不是第一版必须项。

---

### 可选 3：负责人 / 截止日期草稿

系统尝试识别：

- owner
- due date

注意：

该功能容易 hallucination，建议作为实验性能力，而不是核心依赖。

---

### 可选 4：历史记录保存

保存最近几次分析记录。

作用：

提升产品感，但不是闭环所必须。

---

## 6. MVP 明确不做的内容（Out of Scope）

以下内容第一版明确不做，防止范围膨胀。

### 不做 1：登录 / 注册系统

原因：

与核心价值无关，会增加大量额外开发成本。

### 不做 2：多用户协作

原因：

第一版只验证单人使用闭环。

### 不做 3：复杂权限系统

原因：

属于企业级需求，不适合当前阶段。

### 不做 4：真正接入 Jira / Notion / Slack API

原因：

集成成本高，容易拖慢进度。第一版先做导出即可。

### 不做 5：自动发邮件 / 自动建任务

原因：

属于自动执行层，风险高，不是第一阶段重点。

### 不做 6：聊天式 Agent 交互

原因：

容易把项目做成泛聊天工具，偏离“文档到执行”的核心定位。

### 不做 7：复杂向量数据库知识库平台

原因：

第一版重点是单次文档处理，不做完整知识平台。

### 不做 8：音频录音转写

原因：

会增加 ASR 链路复杂度。第一版只处理已有文本。

### 不做 9：多模态输入（图片、截图、视频）

原因：

当前阶段先聚焦纯文本 workflow。

### 不做 10：企业级部署 / 安全 / 审计

原因：

当前项目定位为求职作品和 demo，不是生产级 SaaS。

## 8. MVP 成功标准

### 功能成功标准

用户可以完成一次完整流程：

输入文本

→ 系统生成结构化结果

→ 查看引用依据

→ 修改结果

→ 导出结果

### 输出成功标准

系统能稳定输出以下四类信息：

- Summary
- Action Items
- Risks
- Open Questions

并且格式基本统一、可读。

### 可信度成功标准

至少主要 action items 和 risks 能显示对应原文依据。

### 项目成功标准

这个版本能够：

- 在 3–5 分钟内完成 demo
- 在简历中作为完整项目描述
- 在面试中清楚讲解设计思路与取舍

---

## 9. 第一版开发原则

### 原则 1：先闭环，再扩展

优先做完整流程，不追求大而全。

### 原则 2：先可靠，再智能

宁可结果少一点，也不要乱编很多内容。

### 原则 3：先结构化，再聊天化

重点是输出清晰结果，而不是自由对话体验。

### 原则 4：先人工确认，再自动执行

第一版只做辅助，不做全自动执行。

### 原则 5：先求职价值，再商业复杂度

该项目主要用于展示 Applied AI 能力，不追求第一版商业化完整性。

## 10. MVP+ 与 LLM 岗位工程化（扩展范围）

> **定位**：在**不推翻**上文 Must Have / 成功标准的前提下，为 **LLM 应用 / AI 工程 / Applied AI（偏后端）** 面试增加**可讲述的工程深度**。  
> **执行清单**：见 `开发拆分.md` → **Phase 7：LLM 岗位向 · 企业级工程化补强**。

### 10.1 与「第一版边界」的关系

- **第一版（本文 §3–§8）**：仍定义为「可演示的单人工作台」，功能边界不变。
- **MVP+**：在主线之外，按优先级增加 **鉴权、request_id、结构化日志、限流、CI、密钥规范、API 版本意识** 等；**异步长任务**为可选项。
- **已实现、超出原始 MVP 文本的 LLM 能力**（RAG、LoRA、离线评测、`llm_prompt_chunks`、embedding 缓存等）已在 `开发拆分.md` **§3.1 已实现能力清单** 列出，便于简历对齐。

### 10.2 面试叙事建议（LLM 岗）

- **产品 + 模型**：文档抽取、多模式（rules / LLM / LoRA）、RAG 与指标对比。
- **工程**：为何要做限流与可观测、LLM 超时与降级、密钥与合规意识、CI 防回归。

## 11. 当前版本结论

第一版 MVP 的边界可以总结为一句话：

**做一个单人可用的“文档到执行”AI 工作台，支持结构化提取、引用依据、人工审核和结果导出。**

第一版只验证“信息是否可以顺利转为执行结果”，不验证复杂协作和企业级自动化。**LLM 岗位向的工程化补强**见上文 §10 与 `开发拆分.md` Phase 7。
