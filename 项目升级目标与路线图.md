# Doc2Action 项目升级目标与路线图（面向 Senior 级求职）

> 文档目的：把当前 MVP 升级为“微小型企业级 AI 项目”，并覆盖大模型算法工程师岗位常见能力要求（算法、训练、评估、检索、部署、工程化）。

---

## 1. 最终目标（Target v2）

把 Doc2Action 从“可演示 MVP”升级为“可复现实验 + 可部署服务 + 可量化评估”的项目，最终具备以下交付能力：

1. 多策略分析引擎：规则、API LLM、本地微调模型可切换
2. RAG 检索增强：支持向量检索、重排、引用可追溯
3. 模型训练链路：完成 LoRA/QLoRA 微调实验并可复现
4. 评估体系：有离线指标、对比实验、误差分析
5. 工程化能力：异步任务、缓存、观测、容错、CI
6. Demo 与文档体系：可在 5 分钟演示，也可在面试中讲深度实现

---

## 2. 与 JD 要求的能力映射

### A. NLP / 信息抽取 / 知识检索

- 当前：已完成结构化提取（summary/action/risk/question）
- 升级：增加 NER/关系抽取实验，增加 RAG 检索与重排

### B. LLM 架构与优化

- 当前：API + fallback 规则链路
- 升级：Prompt 模板版本化、策略路由、模型对比与质量门禁

### C. PyTorch / 微调技术（LoRA 等）

- 当前：尚未落地训练
- 升级：建立 `ml/train`，完成 LoRA 训练、推理、评估报告

### D. 数据治理与标注

- 当前：已有样本但无规范标注集
- 升级：建设训练/验证/测试集，制定标注规范与质检规则

### E. 企业级工程能力

- 当前：单机可运行
- 升级：任务队列、缓存、日志追踪、监控指标、压测与稳定性策略

---

## 3. 分阶段升级计划（建议 8-10 周）

## Phase A：基线固化（1 周）

### 目标

把当前 MVP 固化成稳定基线版本，作为后续实验对照组。

### 交付

- 冻结基线接口与输出 schema
- 补齐端到端验收清单
- 形成 `baseline` 报告（当前规则 + API fallback）

### 能力关键词

- 版本管理、基线思维、质量门禁

---

## Phase B：数据与评估体系（1-2 周）

### 目标

把“感觉好不好”变成“指标好不好”。

### 交付

- `ml/data/`：训练/验证/测试数据目录
- `ml/data/schema.json`：标注格式定义
- `ml/eval/evaluate.py`：离线评估脚本
- 指标：
  - Action Precision
  - Risk Coverage
  - Question Usefulness
  - Citation Hit Rate
  - Duplication Rate

### 能力关键词

- 数据清洗、标注规范、评估设计

---

## Phase C：模型微调（2 周）

### 目标

做出可复现的 LoRA 微调链路（PyTorch + Transformers + PEFT）。

### 交付

- `ml/train/train_lora.py`
- `ml/infer/infer_lora.py`
- 训练配置与实验记录（batch size、lr、epoch、loss）
- 基线 vs 微调模型效果对比

### 能力关键词

- PyTorch、LoRA/QLoRA、实验管理

---

## Phase D：RAG 增强（1-2 周）

### 目标

让模型在长文档和业务知识场景下更稳。

### 交付

- 文档切分策略升级（语义分段 + 窗口）
- 向量检索（FAISS/Chroma）
- 检索重排（可选：Cross-Encoder）
- 引用打分与证据质量评估

### 能力关键词

- LangChain 或自建 RAG、向量检索、召回优化

---

## Phase E：服务化与性能（1-2 周）

### 目标

向企业级项目靠拢：可观测、可扩展、可稳定运行。

### 交付

- 后端异步任务（Celery/RQ 二选一）
- Redis 缓存（重复请求去重）
- 结构化日志 + trace id
- 压测脚本与性能报告（P95 延迟、吞吐）
- 失败重试与降级策略文档

### 能力关键词

- 系统设计、稳定性工程、性能优化

---

## Phase F：求职与面试交付（1 周）

### 目标

把技术成果转换为“可投递、可讲解、可验证”的材料。

### 交付

- 技术报告：架构、实验、指标、误差分析
- 面试问答库（算法向、工程向、业务向）
- 5 分钟演示脚本 + 20 分钟深挖脚本
- 简历项目描述（应用版 + 算法版各一版）

### 能力关键词

- 技术表达、抽象能力、结果导向

---

## 4. 最终“微小型企业级”验收标准

## 功能层

- 支持 5 类文档输入并稳定输出结构化结果
- 引用可追溯，且证据命中率 >= 85%
- 支持人工编辑和 Markdown/JSON 导出

## 算法层

- 至少 1 条可复现微调链路（LoRA）
- 至少 1 份离线评估报告（含对比实验）
- 至少 1 份误差分析报告（Top 20 错误类型）

## 工程层

- 服务可连续运行，具备基础监控和异常告警思路
- P95 延迟、吞吐、失败率有明确数据
- 有 CI 检查（lint + tests + sample checks）

## 5. 数据与资源清单（需要你逐步准备）

## 必需资源

1. 语料数据
   - 会议纪要、PRD、邮件、SOP、客服记录（去敏后）
   - 目标规模：首批 300-500 条
2. 标注时间
   - 每天 30-60 分钟，持续 2-3 周
3. 计算资源
   - 本地 CPU 可先跑流程
   - 云 GPU（按需租用）用于微调实验

## 推荐资源

1. 模型与框架
   - PyTorch、Transformers、PEFT、Datasets
   - vLLM（推理优化）
2. 检索与数据层
   - FAISS/Chroma、SQLite/Postgres、Redis
3. 追踪与实验
   - Weights & Biases 或 MLflow（二选一）

## 7. 执行原则

1. 每阶段必须有“代码 + 文档 + 指标”三件套
2. 每次新增能力，都要与基线做对比
3. 先可用，再优化；先可解释，再复杂化
4. 所有实验必须可复现（命令、配置、数据版本）

---

## 8. 你接下来第一步要做什么

从 Phase B 开始：

1. 先建 `ml/data/schema.json`（定义标注字段）
2. 整理首批 50 条标注样本（小步快跑）
3. 搭建 `ml/eval/evaluate.py`，先跑基线评分

完成这三步后，再进入 LoRA 训练阶段，效率最高。
