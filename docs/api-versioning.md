# API 版本与兼容策略（Phase 7）

## 当前约定

- **稳定入口（推荐集成方）**：`/api/v1/health`、`/api/v1/analyze`、`/api/v1/jobs/analyze`、`/api/v1/jobs/{job_id}`。
- **历史路径**：`/health`、`/analyze` 仍保留，便于现有前端与脚本；新集成请优先 **v1**。

## Breaking change（破坏性变更）

- **定义**：删除字段、改字段类型、改语义、改错误码含义等，会导致旧客户端行为变化。
- **策略（本项目 / 小团队适用）**：
  - 尽量 **加字段、不改旧字段**；新语义用新字段或新版本路径（如未来 `/api/v2/...`）。
  - 若必须破坏：在 **CHANGELOG / Release** 中写明迁移步骤；如可能，并行保留旧路由一段时间。

## 非破坏性扩展

- `AnalyzeResponse.meta` 中增加键：默认视为安全（客户端应忽略未知键）。
- 新增可选请求体字段：需有默认值或「缺省与旧行为一致」。

## OpenAPI

- 服务启动后访问 `/docs` 查看当前契约；版本迭代时同步检查 **schema diff**。
