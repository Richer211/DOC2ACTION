## 项目名称

**Doc2Action AI Workspace**

中文暂定名：**文档到执行工作台**

---

## 1. 项目一句话定义

一个将**会议纪要、需求文档、邮件、SOP 等非结构化文本**，自动转化为**摘要、行动项、风险点、待确认问题**的 AI 工作台，并支持**引用原文、人工审核、导出结果**。

---

## 2. 项目背景

在真实工作中，团队每天会接触大量非结构化信息，例如：

- 会议纪要
- PRD / 需求文档
- 邮件沟通
- SOP / 操作说明
- 客服记录
- 临时备注

这些信息通常需要人工整理后，才能变成真正可执行的内容，例如：

- 下一步要做什么
- 谁负责
- 优先级如何
- 风险点是什么
- 哪些问题还没有结论

目前这个过程通常存在几个问题：

1. **人工整理耗时**
2. **容易遗漏 action items**
3. **多人理解不一致**
4. **文档和执行之间断层**
5. **缺少统一、可追溯的结果输出**

因此，需要一个 AI 工作台，把“信息”直接推进到“执行”。

---

## 3. 项目目标

本项目的目标不是做一个通用聊天机器人，而是做一个**面向工作流的 AI 工具**。

### 核心目标

让用户在上传或粘贴一段文档后，能够快速获得：

- 摘要 summary
- 行动项 action items
- 风险点 risks
- 待确认问题 open questions
- 引用来源 source evidence
- 可编辑、可导出的最终结果

### 项目目标关键词

- 结构化输出
- 文档到执行
- AI + human review
- 可追溯
- 可演示
- 可写入简历 / 可用于求职展示

## 4. 目标用户

### 第一阶段目标用户

适合处理文档、会议、任务的人群：

- 产品经理
- 小团队负责人
- 工程师

- 创业团队成员

---

## 5. 用户核心痛点

### 痛点 1：信息很多，但执行结果不清晰

用户看完会议纪要或文档后，仍然需要自己再整理一遍。

### 痛点 2：容易漏项

例如某个 action item、一个隐含风险、一个未决问题，很容易在人工整理时被忽略。

### 痛点 3：输出不统一

不同人整理风格不同，导致后续协作成本高。

### 痛点 4：结果不可追溯

很多 AI 工具只能给一个 summary，但不能说明“这条任务是从哪段原文来的”。

### 痛点 5：AI 不能直接替代人

如果 AI 直接生成结果但不可编辑、不可确认，用户不敢真的拿去执行。

## 6. 核心使用场景

### 场景 1：会议纪要整理

用户上传一份会议纪要，系统输出：

- 会议摘要
- 下一步行动项
- 风险点
- 待确认问题

### 场景 2：PRD / 需求文档整理

用户上传 PRD，系统输出：

- 核心需求总结

- 依赖项
- 风险点

### 场景 3：邮件 / 沟通记录整理

用户粘贴邮件链，系统输出：

- 结论摘要
- 待办事项
- follow-up 点
- 未解决问题

## 7. MVP 核心功能

第一版只做最小闭环。

### 功能 1：文本输入 / 文档输入

支持：

- 手动粘贴文本
- 上传 `.txt` / `.md`
- 后续可扩展 PDF / DOCX

### 功能 2：文档解析

系统对输入内容进行基础处理：

- 分段
- chunk 切分
- 编号
- 清洗空行/噪声

### 功能 3：结构化提取

系统调用 LLM 输出以下模块：

- Summary
- Action Items
- Risks
- Open Questions

### 功能 4：引用回溯

每条 action item / risk / question 尽量能关联原文片段，供用户查看依据。

### 功能 5：人工审核编辑

用户可以：

- 修改 summary
- 编辑/删除/新增 action item
- 修改优先级
- 确认最终结果

### 功能 6：结果导出

支持导出：

- Markdown
- JSON
- CSV（可选）

---

## 8. MVP 明确不做什么

这是最重要的一部分之一。

### 第一版不做：

- 用户登录系统
- 多用户协作
- 权限系统
- 真正接入 Jira / Notion / Slack API
- 自动发邮件
- 多轮聊天助手
- 复杂 agent 自主执行
- 向量数据库全平台知识库
- 长期历史记忆
- 多模态音频转录
- 企业级权限 / 审计 / 合规机制

### 原则

第一版不是 SaaS 产品，不追求完整商业化，只追求：

**一个清晰、稳定、能演示的 AI workflow 闭环。**

## 12. 项目主要风险

### 风险 1：LLM 输出不稳定

同类文档可能出现不同风格输出，或者漏提任务。

### 风险 2：幻觉

模型可能编造负责人、截止日期、结论。

### 风险 3：引用映射困难

输出结果与原文片段无法准确绑定，会影响可信度。

### 风险 4：边界失控

容易不断加功能，导致开发周期拉长。

### 风险 5：做成“普通摘要工具”

如果没有结构化输出、审核和导出，就会显得太弱。
